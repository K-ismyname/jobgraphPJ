# 에이전트 v3 설계 — 다중 소스 적합도 평가 (Multi-Source Fit Assessment)

> 작성일: 2026-06-11
> 대상: 평가를 **적합도(직무를 얼마나 잘하나) + 신뢰도(얼마나 확실한가) 두 축**으로 분리하고,
>       이력서·포트폴리오·GitHub·배포URL 4개 소스를 독립 평가자로 합쳐 판단
> 관계: v2(슬림 4노드, 검증 중심)를 발전. Gap/Critic/Coach는 유지, Profile을 다중 평가자로 확장.

---

## 1. 진화 배경

| 버전 | 핵심 | 문제 |
|------|------|------|
| v1 | 7노드 Plan-and-Execute | 과설계 (Market 데드, Retrieval 중복, Planner 장식) |
| v2 | 4노드 슬림 + Critic 등급화 | 멀티에이전트가 단순 파이프라인이라 "억지" |
| **v3** | **다중 소스 적합도 평가** | 기능 자체가 멀티에이전트를 요구 → 억지 해소 |

세 가지 깨달음이 v3를 만들었다:
1. **멀티에이전트는 기능에서 나와야 한다** — "여러 독립 소스를 각자 평가 → 합의"는 멀티에이전트가 자연스럽게 필요한 작업.
2. **검증은 수단, 적합도가 목적** — "자료가 진짜냐"가 아니라 "직무를 얼마나 잘하느냐".
3. **적합도와 신뢰도는 별개 축** — 이력서만으로도 적합 판단은 한다. 검증은 그 판단의 신뢰도를 높일 뿐, 적합 판단의 전제가 아니다.

---

## 2. 핵심 원칙

1. **적합도(Fit) ⊥ 신뢰도(Trust) — 두 축 분리**
   - **적합도**: "스킬이 직무 요구 수준에 맞나?" → **이력서만으로도 판단**
   - **신뢰도**: "그 판단이 얼마나 확실한가?" → 증거 소스가 많을수록 ↑
   - 검증 부족 ≠ 적합도 낮음. 검증 부족 = **신뢰도** 낮음 (적합 판단은 함).
2. **적합도 = 수준 매칭** — "보유 여부"가 아니라 "요구 수준만큼 다루는가". 갭 = 수준 차이.
3. **판단하는 것만 에이전트** — 각 소스 평가자는 독립 판단 주체. 검색·조회는 도구.
4. **증거 노출** — 임의로 단정하지 않고 "어느 소스에 무슨 증거"를 보여주고 사용자가 판단.
5. **교차 검증** — 여러 독립 소스가 일치하면 신뢰↑ (GitHub 없어도 작동).

---

## 3. 4개 평가자 (각자 다른 modality·적합도 측면)

| 평가자 | 입력 | modality | 무엇을 보나 (적합도 측면) |
|--------|------|----------|--------------------------|
| **이력서** | PDF/텍스트 | 텍스트 | 경력 기간·직책·도메인 — "얼마나 오래/어떤 환경" |
| **포트폴리오** | PDF(이미지+텍스트) | **vision (멀티모달)** | 프로젝트 규모·역할·성과 — "어떤 맥락/규모로" |
| **GitHub** | URL | 코드 분석 | 코드 복잡도·패턴·테스트 — "얼마나 잘 짜나" |
| **배포 URL** | URL | **웹 분석 (HTML+스크린샷 vision)** | 작동 실증·완성도·배포 경험 — "실제로 만들어 운영했나" |

각 평가자가 **완전히 다른 modality**라 한 LLM에 합칠 수 없다 → 멀티에이전트 정당성의 핵심.
**입력은 모두 선택적** — 있는 소스만 평가. 이력서만 줘도 적합도 판단은 되고, 소스가 많을수록 신뢰도가 올라간다.

---

## 4. 평가자별 상세

### 4-1. 포트폴리오 평가자 (멀티모달)
- **PDF→이미지**: PyMuPDF `get_pixmap(dpi=220)` (poppler 의존성 없음)
- **텍스트**: pdfplumber (vision OCR 보정용)
- **이해**: GPT-4o vision, `detail="high"`, `temperature=0`, structured output
- **페이지별 호출** 후 프로젝트 단위 병합
- **프롬프트 핵심**: (a) 프로젝트 단위 구분 (b) 다이어그램의 박스·화살표·라벨 명시 분석 (c) 근거 출처(text/diagram/screenshot) 기록

### 4-2. 배포 URL 평가자 (웹 분석)
- **HTML/소스**: WebFetch로 프론트 기술 단서(`__NEXT_DATA__` 등)
- **스크린샷+vision**: headless 브라우저 캡처 → UI 완성도·작동 판단
- **한계(정직)**: 프론트·작동·완성도는 보이나 **백엔드·AI 기술(LangGraph 등)은 외부에서 안 보임**. 주 가치는 "작동 실증 + 배포 경험"(타겟 공고 'PoC→MVP→배포' 요구 직결)
- GitHub와 교차: "코드 있음 + 작동함" = 강한 검증

---

## 5. 합의 노드 (Map-Reduce, Supervisor 없음) — "서기" 역할

4 평가자는 병렬 fan-out(map) → 합의 노드(reduce). 중앙 동적 라우터(Supervisor)는 불필요(흐름 고정). 디스패처는 결정적(입력에 있는 소스만 Send).

**원칙: 합의 노드는 "판단"이 아니라 "사실 종합"** — 법정의 *서기*(증거 정리·기록)이지 *판사*(판결)가 아니다. 검증 상태 판정 + 증거 통합만 하고, 적합도·숙련도 판단은 Gap에 위임 → 합의 노드에 임의성이 없다.

### 동작 — 4단계 (전부 결정적)
```
입력: 4 평가자의 [{skill, evidence, source, level_hint}]

① 스킬 정규화·매칭   normalize_skill로 통일 → 같은 스킬 묶기
② 검증 상태 판정 (결정적, 임의 점수 없음)
     GitHub/배포 증거    → Verified
     2+ 소스 일치        → Corroborated
     1 소스만            → Claimed
     없음(요구 스킬)      → Absent
③ 증거 통합           모든 소스 증거를 목록으로
④ 충돌 플래그         "주장 O + 코드 X" → "코드 미확인" 플래그
                     (숙련도는 단정 안 함 — Gap이 판단)
```

### 출력 스키마
```python
{
  "LangGraph": {
    "verification": "Verified",
    "evidences": [
      {"source": "github",    "text": "StateGraph import + 노드 정의", "level_hint": "실무"},
      {"source": "portfolio", "text": "멀티에이전트 RAG 주도",          "level_hint": "실무"},
      {"source": "resume",    "text": "LangGraph로 구축",              "level_hint": null}
    ]
  },
  "Docker": {
    "verification": "Claimed",
    "evidences": [{"source": "resume", "text": "Docker 사용", "level_hint": null}],
    "flags": ["코드 미확인 — 이력서 주장만"]
  }
}
```

> **검증 상태 = 신뢰도 축.** "검증됐나"는 사실 판정이지 임의 점수가 아니다. "여러 독립 소스 일치 = 신뢰↑"는 법정·저널리즘의 원칙. GitHub 없는 사람도 교차검증(Corroborated)으로 차별화.

---

## 6. Gap (적합도 판단) — "판사" 역할

합의가 정리한 사실(검증상태 + 증거)을 받아 **적합도를 판단**한다. **이력서만 있어도(주장 기반) 판단한다** — 검증은 전제가 아니라 신뢰도.

```
LangGraph: 보유=실무(주장, 일부 검증) vs 직무 요구=프로덕션
  → 적합하나 "프로덕션 규모 경험" 갭
Docker:    보유=언급만(Claimed) vs 직무 요구=실무
  → 주장은 있으나 근거 약함
```
- 숙련도 척도: **기초/실무/전문** (증거 기반 + 보조 라벨, 척도 임의성은 증거 노출로 완화)
- 직무 요구 수준: 공고 텍스트에서 추출 ("X년", "프로덕션", "설계" 등 단서)
- **갭 = 보유 수준과 요구 수준의 차이** (binary 아님)

## 7. 두 축 출력 + 검증 조언 (사용자 대면)

최종 리포트는 **적합도 + 신뢰도 + 조언**을 함께 보여준다:

```
직무 적합도: 72%   (필수 스킬 8개 중 6개를 요구 수준으로 보유)
신뢰도: ◐ 보통    (이력서 주장 기반, 미검증)

💡 GitHub·포트폴리오를 추가하면 코드·프로젝트로 검증해
   더 정확한 분석이 가능합니다.

스킬별:
  LangGraph [요구=실무 / 보유=실무(주장) / Verified ✓] — 적합, 검증됨
  Docker    [요구=실무 / 보유=언급만 / Claimed ◐]       — 적합하나 근거 약함
```

**적합도는 이력서만으로도 산출**하고, 신뢰도(검증 수준)와 "더 정확히 하려면 무엇을 주라"는 조언을 함께 붙인다. → 진입 장벽 낮음 + 자연스러운 증거 유도.

### 축별 역할 분담
| 노드 | 담당 축 |
|------|---------|
| 합의 노드 | **신뢰도** (검증 상태 Verified/Claimed…) |
| Gap | **적합도** (직무 요구 vs 보유 수준, 주장 포함) |
| Coach | 갭 개선 제안 + **"검증 강화하려면 GitHub/포폴 추가"** 조언 |

## 8. 그래프 구조

```
START
  ▼
디스패처 (결정적: 입력에 있는 소스만 Send)
 ┌──────────┬───────────┬──────────┬───────────┐
 ▼          ▼           ▼          ▼
이력서      포폴         GitHub     배포URL      (병렬 map)
평가자      평가자       평가자     평가자
 └──────────┴─────┬─────┴──────────┘
                  ▼
            합의 노드 (reduce — 검증상태 + 교차검증 + 증거, "서기")
                  ▼
            Gap (적합도: 보유 수준 vs 직무 요구, "판사")
                  ▼
            Critic (갭 주장의 근거 충실성 검증, 길 B 등급화)
                  ▼
            Coach (근거 기반 개선 제안 + 검증 조언)
                  ▼
                END
```
Supervisor 없는 Map-Reduce 멀티에이전트. Gap/Critic/Coach는 v2 그대로(내부 도구 루프 유지).

## 9. 멀티에이전트 정당성
- **다른 modality** (텍스트/vision/코드/웹) — 한 LLM에 못 합침
- **독립성** — 각 평가자가 다른 소스에 안 휘둘림
- **병렬(Send)** + **교차 검증** — 여러 독립 평가의 합의 = 신뢰↑
- 기능이 멀티에이전트를 *요구* → "억지" 해소

---

## 10. 한계 (정직)
- **배포 URL**: 백엔드·AI 기술은 외부에서 안 보임 → 주 가치 "작동 실증·배포 경험"
- **숙련도 척도**: 단계 구분에 일부 임의성 → 증거 노출로 완화, 데이터로 튜닝
- **vision 비용**: 포폴 페이지·스크린샷마다 호출 (비용 무관 전제)
- **소스 1개(이력서만)**: 적합도는 판단하되 신뢰도 "보통" + 검증 조언

## 12. 구현 순서 (writing-plans에서 상세화)

1. **텍스트 MVP** — 이력서·GitHub 평가자 + 합의(검증상태·교차검증) + Gap 적합도. 멀티모달 없이 동작
2. 포폴 평가자(멀티모달 vision) 추가
3. 배포 URL 평가자 추가
4. 두 축 출력(적합도+신뢰도) + 검증 조언 + Critic/Coach 연결
5. end-to-end

> **단계 1(텍스트 MVP)이 핵심** — 어려운 멀티모달·웹 분석을 처음부터 안 떠안고, 텍스트 소스로 합의·적합도 구조부터 동작시킨 뒤 modality를 하나씩 얹는다.

## 13. 포트폴리오 서사

"단일 이력서 분석 → 다중 소스 교차검증으로 진화. **적합도(맞나)와 신뢰도(확실한가)를 두 축으로 분리**하고, 각 소스를 다른 modality의 독립 평가자로 합의. 멀티에이전트가 기능에서 자연 발생." — "억지 멀티에이전트"를 "기능이 요구하는 멀티에이전트"로 만든 의사결정 과정이 핵심.
