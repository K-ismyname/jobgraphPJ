# 에이전트 v3 설계 — 다중 소스 적합도 평가 (Multi-Source Fit Assessment)

> 작성일: 2026-06-11
> 대상: 평가의 축을 "스킬 보유 여부 + 진위 검증"에서 **"직무 적합도(숙련도 수준 매칭)"**로 전환하고,
>       이력서·포트폴리오·GitHub·배포URL 4개 소스를 독립 평가자로 합쳐 판단
> 관계: v2(슬림 4노드, 검증 중심)를 발전. Gap/Critic/Coach는 유지, Profile을 다중 평가자로 확장.

---

## 1. 진화 배경

| 버전 | 핵심 | 문제 |
|------|------|------|
| v1 | 7노드 Plan-and-Execute | 과설계 (Market 데드, Retrieval 중복, Planner 장식) |
| v2 | 4노드 슬림 + Critic 등급화 | 멀티에이전트가 단순 파이프라인이라 "억지" |
| **v3** | **다중 소스 적합도 평가** | 기능 자체가 멀티에이전트를 요구 → 억지 해소 |

**두 가지 깨달음이 v3를 만들었다:**
1. **멀티에이전트는 기능에서 나와야 한다** — 단순 기능에 욱여넣으면 억지. "여러 독립 소스를 각자 평가 → 합의"는 멀티에이전트가 *자연스럽게* 필요한 작업.
2. **검증은 수단, 적합도가 목적** — "자료가 진짜냐(검증)"가 아니라 "이 사람이 직무를 얼마나 잘하느냐(적합도/숙련도)"가 핵심.

---

## 2. 핵심 원칙

1. **적합도 > 검증** — "스킬 보유 여부"가 아니라 "요구 수준만큼 다루는가". 갭 = 수준 차이.
2. **판단하는 것만 에이전트** — 각 소스 평가자는 독립 판단 주체. 검색·조회는 도구.
3. **증거 노출** — confidence를 임의로 단정하지 않고, "어느 소스에 무슨 증거"를 보여주고 사용자가 판단.
4. **교차 검증** — 여러 독립 소스가 일치하면 신뢰↑ (GitHub 없어도 작동).

---

## 3. 4개 평가자 (각자 다른 modality·적합도 측면)

| 평가자 | 입력 | modality | 무엇을 보나 (적합도 측면) |
|--------|------|----------|--------------------------|
| **이력서** | PDF/텍스트 | 텍스트 | 경력 기간·직책·도메인 — "얼마나 오래/어떤 환경" |
| **포트폴리오** | PDF(이미지+텍스트) | **vision (멀티모달)** | 프로젝트 규모·역할·성과 — "어떤 맥락/규모로" |
| **GitHub** | URL | 코드 분석 | 코드 복잡도·패턴·테스트 — "얼마나 잘 짜나" |
| **배포 URL** | URL | **웹 분석 (HTML+스크린샷 vision)** | 작동 실증·완성도·배포 경험 — "실제로 만들어 운영했나" |

각 평가자가 **완전히 다른 modality**라 한 LLM에 합칠 수 없다 → 멀티에이전트 정당성의 핵심.

**입력은 모두 선택적** — 있는 소스만 평가(디스패처가 결정적으로 Send). 이력서만 줘도 동작하고, 소스가 많을수록 신뢰도가 올라간다("증거 많을수록 신뢰↑").

---

## 4. 평가자별 상세

### 4-1. 포트폴리오 평가자 (멀티모달)
- **PDF→이미지**: PyMuPDF `get_pixmap(dpi=220)` (poppler 의존성 없음)
- **텍스트**: pdfplumber (vision OCR 보정용)
- **이해**: GPT-4o vision, `detail="high"`, `temperature=0`, structured output
- **페이지별 호출** 후 프로젝트 단위 병합 (한 번에 다 주면 뒷 페이지 흐려짐)
- **프롬프트 핵심**: (a) 프로젝트 단위 구분 (b) 다이어그램의 박스·화살표·라벨 명시 분석 (c) 근거 출처(text/diagram/screenshot) 기록

### 4-2. 배포 URL 평가자 (웹 분석)
- **HTML/소스**: WebFetch로 프론트 기술 단서(`__NEXT_DATA__` 등)
- **스크린샷+vision**: headless 브라우저 캡처 → UI 완성도·작동 판단
- **한계(정직)**: 프론트 기술·작동·완성도는 보이나, **백엔드·AI 기술(LangGraph 등)은 외부에서 안 보임**. 주 가치는 "작동 실증 + 배포 경험"(타겟 공고 'PoC→MVP→배포' 요구 직결)
- GitHub와 교차: "코드 있음(GitHub) + 작동함(URL)" = 강한 검증

---

## 5. 합의 (Map-Reduce, Supervisor 없음)

4 평가자는 **병렬 fan-out(map)** → **합의 노드(reduce)**. 중앙 동적 라우터(Supervisor)는 불필요 — 흐름이 고정이라 라우팅할 게 없다. 디스패처는 결정적(입력에 있는 소스만 Send), 합의 노드는 종합.

### 검증 상태 (점수제 대신 — 임의성 제거)
스킬마다 **객관적 검증 상태**를 판정한다 (임의 가중치 없음):

| 상태 | 판정 |
|------|------|
| **Verified** ✓ | GitHub 코드·배포 실물로 실증 |
| **Corroborated** ◎ | 여러 소스가 일치 (이력서+포폴 같은 프로젝트) — GitHub 없어도 가능 |
| **Claimed** ◐ | 한 소스에만 언급 |
| **Absent** ✗ | 어디에도 없음 |

"검증됐나"는 사실 판정이지 임의 점수가 아니다. "여러 독립 소스 일치 = 신뢰↑"는 법정·저널리즘의 원칙. → GitHub 없는 사람도 교차검증(Corroborated)으로 차별화.

### 증거 노출 (단정 대신 사실)
등급을 단정하지 않고 증거 목록을 함께 노출:
```
LangGraph [Verified ✓ / 실무 수준]
  - GitHub: StateGraph import + 노드 정의 코드
  - 포폴: 멀티에이전트 RAG 프로젝트 주도 (다이어그램)
  - 이력서: "LangGraph로 구축" (주장)
```
임의 라벨이 결론을 지배하지 않고, 사용자가 증거로 직접 판단. = "왜 믿을 수 있나"의 정직한 답.

---

## 6. 적합도 판단 — 숙련도 수준 vs 직무 요구

합의로 나온 종합 스킬 프로필을 **수준**으로 평가하고, **직무 요구 수준**과 대조한다:

```
종합 숙련도: "LangGraph 실무 수준" (GitHub 코드 + 포폴 프로젝트로 검증)
직무 요구:   "프로덕션 멀티에이전트 경험"
  → 갭: "프로덕션 규모·안정성 경험 부족" (없음이 아니라 수준 차이)
```

- 숙련도 척도: **기초 / 실무 / 전문** (증거 기반 + 보조 라벨, 척도 임의성은 증거 노출로 완화)
- 직무 요구 수준: 공고 텍스트에서 추출 ("X년 경력", "프로덕션", "설계 경험" 등 단서)
- **갭 = 보유 수준과 요구 수준의 차이** (binary 아님)

## 7. 그래프 구조

```
START
  │
  ▼
디스패처 (결정적: 입력에 있는 소스만 Send)
 ┌──────────┬───────────┬──────────┬───────────┐
 ▼          ▼           ▼          ▼
이력서      포폴         GitHub     배포URL      (병렬 map)
평가자      평가자       평가자     평가자
(텍스트)   (vision)    (코드)     (웹)
 └──────────┴─────┬─────┴──────────┘
                  ▼
            합의 노드 (reduce — 검증상태 + 교차검증 + 증거 종합)
                  ▼
            Gap (숙련도 vs 직무 요구 수준 → 갭)
                  ▼
            Critic (갭 주장의 근거 충실성 검증, 길 B 등급화)
                  ▼
            Coach (근거 기반 이력서 개선 제안)
                  ▼
                END
```

Supervisor 없는 Map-Reduce 멀티에이전트. Gap/Critic/Coach는 v2 그대로 (내부 도구 루프 유지).

## 8. 멀티에이전트 정당성

- **다른 modality** — 텍스트/vision/코드/웹, 한 LLM에 못 합침
- **독립성** — 각 평가자가 다른 소스에 안 휘둘림 (편향 방지)
- **병렬(Send)** — 동시 평가
- **교차 검증** — 여러 독립 평가의 합의 = 신뢰↑
- 기능(다중 소스 적합도 평가)이 멀티에이전트를 *요구* → "억지" 해소

---

## 9. 한계 (정직)

- **배포 URL**: 백엔드·AI 기술은 외부에서 안 보임 → 주 가치는 "작동 실증·배포 경험"
- **숙련도 척도**: 어떤 단계 구분이든 일부 임의 → 증거 노출로 완화, 데이터로 튜닝
- **vision 비용**: 포폴 페이지·스크린샷마다 호출 (비용 무관 전제로 진행)
- **소스 1개(이력서만)**: 교차검증·Verified 못 받음 → "GitHub·배포 주면 강화" 유도

## 11. 구현 순서 (writing-plans에서 상세화)

1. 텍스트 평가자부터 (이력서/GitHub) + 합의(검증상태·교차검증) — 멀티모달 없이 동작하는 MVP
2. 포폴 평가자(멀티모달 vision) 추가
3. 배포 URL 평가자 추가
4. 적합도(숙련도 수준) 평가 + 직무 요구 수준 대조
5. Gap/Critic/Coach 연결 + end-to-end

> **단계 1(텍스트 MVP)이 핵심** — 멀티모달·웹 분석은 어려운 부분이니, 텍스트 소스로 합의 구조부터 동작시킨 뒤 modality를 하나씩 얹는다.

## 12. 포트폴리오 서사

"단일 이력서 분석 → 다중 소스(이력서·포폴·GitHub·배포) 교차검증으로 진화. 검증(진위)에서 적합도(숙련도 수준 매칭)로 축을 옮기고, 각 소스를 다른 modality의 독립 평가자로 합의. 멀티에이전트가 기능에서 자연 발생." — "억지 멀티에이전트"를 "기능이 요구하는 멀티에이전트"로 만든 의사결정 과정이 핵심.
