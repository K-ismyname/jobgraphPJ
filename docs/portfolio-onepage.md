# JobGraph — 포트폴리오 1페이지

> 채용공고를 분석하고 이력서·GitHub·배포를 교차검증해 **직무 적합도**와 그 판단의 **신뢰도**를 분리 산출하는 Agentic RAG 시스템.

---

## [표지]

**JobGraph — 다중 소스 교차검증 Agentic RAG 직무 적합도 분석기**
적합도 정확도 **30% → 83%** 개선 · 공고 **560개 / 9개 직군** · 평가지표(RAGAS) 직접 구축

- **기간**: 2026.06 ~ 현재 (약 1개월)
- **팀 구성**: 개인 프로젝트 (기획·설계·구현·배포 전담)
- **배포**: https://gaheelee-job-graph.hf.space (HF Spaces, 라이브 데모)
- **저장소**: https://github.com/K-ismyname/jobgraphPJ

**기술 스택 (카테고리별)**
- **언어/런타임**: Python
- **에이전트**: LangGraph (StateGraph, `Send` 병렬 fan-out, 조건 분기·루프·HITL)
- **LLM**: OpenAI gpt-4o-mini / gpt-4o (+ vision)
- **그래프 DB**: Neo4j Aura (직군–스킬 관계: REQUIRES / PREFERS / CO_OCCURS / PART_OF)
- **데이터**: Adzuna API
- **문서 처리**: pdfplumber(텍스트) · PyMuPDF(이미지→vision)
- **평가/관측**: RAGAS(faithfulness·answer relevancy) · Langfuse(트레이싱)
- **서빙/배포**: FastAPI + Docker · HF Spaces(Docker SDK) · 정적 웹 프론트(HTML/CSS/JS, Mermaid.js)
- **품질**: pytest (단위 187 + 통합 14)

## [①②] 프로젝트 소개 + 배경/문제

**한 줄 설명**
이력서·GitHub·배포 URL을 종합해 "이 사람이 이 직무에 맞는가"를 **적합도(가졌나)** 와 **신뢰도(검증됐나)** 두 축으로 분리해 답하는 Agentic RAG 시스템.

**타겟 사용자 + 상황**
- 직무 전환·이직을 준비하는 개발자, 그리고 지원자를 빠르게 선별해야 하는 채용 담당자.
- 채용공고 **560개**를 수집해 **9개 직군**(AI/LLM·ML·Data Scientist/Engineer/Analyst·Software·Frontend·DevOps/SRE·Security)의 핵심 스킬 지형을 구축.

**기존 서비스 대비 차별점**
- 기존 도구는 적합도를 **한 점수로 뭉갠다** — "스킬 많은 사람"이 "검증된 사람"처럼 부풀려진다.
- JobGraph는 두 질문을 분리한다: ① 요구 스킬을 **가졌나(적합도)**, ② 그게 **코드·배포로 검증됐나, 이력서 주장일 뿐인가(신뢰도)**.
- 여러 독립 소스(이력서·GitHub·배포)를 **교차검증**해 신뢰도를 **결정적으로** 산출 — LLM의 임의 점수가 아니라 코드가 계산.

- FastAPI 서버 + 웹 프론트 + Docker 컨테이너화 + HF Spaces 배포

## [시각화] 아키텍처 다이어그램

```
START
  └─ dispatcher : 입력에 있는 소스만 Send로 fan-out (병렬)
        ├─ resume_eval     (이력서 텍스트)
        ├─ github_eval     (repo 언어·README·의존성 + LLM 프로젝트 프로필)
        ├─ deploy_eval     (배포 URL HTTP 헤더·콘텐츠)
        └─ portfolio_eval  (PDF → vision 분석)
  → consensus    : 소스별 증거를 검증 등급으로 결정적 판정
                   (Verified=실증 / Corroborated=2+ 출처 / Claimed=주장)
  → Gap 루프     : call_model ↔ tools (Corrective RAG)
                   증거 부족 시 그래프·통계 도구를 스스로 재검색
  → synthesizer  : 적합도 + 신뢰도 (수치는 코드가 결정적 계산)
  → critic       : 리포트를 합의(사실)와 대조 → 환각 제거·라벨 교정
  → coach        : GitHub 프로젝트 기반 "무엇을 더하면 부족 스킬을 실증하나" 코칭
  → END
```
> 소스마다 modality(텍스트·코드·웹·이미지)가 달라 **전용 평가자**로 분리 — 멀티에이전트가 기능적으로 필요한 구조.

## [④⑥] 해결 과정 (상황 → 해결 → 결과)

**문제 1 — 적합도가 비현실적으로 낮게 나옴**
- 상황: 실제 대기업 합격자 이력서도 적합도 **30%**로 과소평가됐다.
- 원인 분석: 스킬을 거친 "역량"으로 묶어 평균을 내니, 폭넓은 역량 중 일부만 가진 사람이 전부 깎였다(평균 편향).
- 해결: 역량 평균이 아니라 **직군 핵심 스킬 개별 비교(스킬 레벨)** 로 적합도를 재정의.
- 결과: 적합도 **30% → 83%** 로 정상화. 직군 변별력도 확보(예: Data Analyst↔Tableau·Power BI·Excel, Data Engineer↔Snowflake·Spark·Scala로 뚜렷이 구분).

**문제 2 — 벡터 검색(Chroma)이 정말 기여하는가**
- 상황: "RAG니까 벡터 DB는 당연히 있어야지"라는 가정으로 Chroma를 넣었다.
- 해결: 기여도를 직접 측정 — 검색 **12건 중 11건**이 단순 키워드 매칭과 **동일한 결과**.
- 결과: 추가 효과가 거의 없음을 확인하고 **Chroma 제거**, 공고 원문을 Neo4j 그래프에 통합. "있으면 좋아 보여서"가 아니라 **효과로 컴포넌트를 판단**.

**문제 3 — 직군 데이터 오염**
- 상황: "Architect"가 SAP·Data·Solution 아키텍트 잡탕으로 묶이고, Security 직군 특화 스킬이 약했다.
- 해결: 직군 분류 키워드와 수집 쿼리를 정교화(SIEM·침해대응·취약점 등 보안 키워드 보강), 잡탕 직군 제거.
- 결과: Security 공고 **15 → 107개(7배)**, 직군별 핵심 스킬이 그 직군 정체성과 일치.

**문제 4 — "왜 이 답을 신뢰할 수 있는가"를 수치로 증명**
- 상황: RAG 응답이 근거에 기반하는지(환각 여부)를 측정할 장치가 없었다.
- 해결: RAGAS로 평가 파이프라인 구축 — 무관 근거 필터링 + 응답을 검증 가능한 사실 진술로 정규화.
- 결과: **answer relevancy 0.36 → 0.90**, **faithfulness 0 → 0.167**(측정 복구). 이후 근거 풍부화로 **0.167 → 0.200**.
- (가설 검증) faithfulness를 더 올리려 응답 표현을 바꿔봤으나 **측정 결과 악화(0.200 → 0)** → 직감 대신 측정을 따라 **롤백**. "가설 2개를 측정해 개선된 것만 채택"한 사례.

**문제 5 — 스킬 표기 분산**
- 상황: `machine learning`/`Machine Learning`, `ML`/`Machine Learning`이 따로 집계돼 통계가 쪼개졌다.
- 해결: 정규화 함수에 표기·동의어 통합(`MLOps`·`Power BI`·`GitHub`·`Cloud Security` 등 브랜드/약어 사전).
- 결과: 동일 개념이 하나로 합쳐져 직군 핵심 스킬 통계가 정확해짐.

**문제 6 (예상 못 한 이슈) — 재적재 일부 누락**
- 상황: Neo4j 적재가 **248/535**로 일부만 들어가 원인 불명이었다.
- 해결: 명령의 `tail`에 오류 로그가 잘려 안 보였던 것 + Neo4j Aura **무료 티어의 일시(transient) 오류**임을 재현으로 진단.
- 결과: 코드 버그가 아님을 확인, 재실행으로 **560/560** 정상 적재. (현상이 아니라 원인까지 추적)

## [⑧] 회고

**잘한 점**
- 추측이 아니라 **측정으로 결정**했다 — Chroma를 효과 측정 후 제거하고, RAGAS 개선 가설도 측정으로 채택/기각했다.
- "적합도"와 "신뢰도"를 **두 축으로 분리**한 설계가 차별점이 됐고, 신뢰할 값은 LLM이 추측하지 않고 **코드가 결정적으로 계산**해 환각을 차단했다.
- 현상에서 멈추지 않고 **원인까지 추적**했다(적합도 30%의 원인=평균 편향, 적재 누락의 원인=transient).

**아쉬운 점**
- 연봉 데이터 공개율이 **3%**에 그쳐 연봉 분석을 보조 지표로 격하해야 했다 — 데이터 소스 한계를 늦게 확인했다.
- RAGAS faithfulness가 합성 단문 평가의 특성상 **~0.2에서 한계**였다 — 평가 설계를 처음부터 더 정교히 했어야 했다.
- 공고 데이터에 LangChain 명시가 적어 타겟 직군 핵심 스킬에 LangChain이 상위로 드러나지 않았다.

**다시 한다면 / 다음 프로젝트 연결**
- 데이터 소스를 다변화(연봉·LangChain 커버리지 확보)하고, **평가 지표를 설계 초기에 먼저 정의**한 뒤 기능을 붙이겠다.
- 다음으로는 **평가 파이프라인을 상시 관측(observability)에 통합**하고, 사용자 데이터가 쌓이면 **추출 모델 파인튜닝(QLoRA)** 으로 비용·정확도를 함께 잡는 방향으로 확장하겠다.
